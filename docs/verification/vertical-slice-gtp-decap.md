## 2. 详细规格定义

### 2.1 RR (Raw Requirement) - 原始需求
**编号**: `RR.UPF.FUN.001`
**描述**: UE 可以通过 N3 接口上的 GTP-U 隧道访问 DN。
**规格逻辑 (Background-Status-Target)**:
- **背景 (Background)**: 5G 核心网用户面功能 (UPF) 作为数据面核心，必须能够处理来自接入网 (gNodeB) 的 N3 接口流量，这是 UE 访问外部数据网络的必经路径。
- **现状 (Status)**: 目前系统尚不支持 N3 接口的终结功能，无法解析 GTP-U 隧道封装的报文，导致上行流量无法被正确解封装并转发至 DN。
- **目标 (Target)**: 实现 N3 接口终结功能，支持对 GTP-U 隧道报文的解析和解封装，使 UE 能够通过 N3 隧道透明访问外部数据网络 (DN)。

### 2.2 IR (Internal/Feature Requirement) - 特性需求
**编号**: `IR.UPF.FUN.001.001`
**特性描述**: **高性能 GTP-U 解封装**
- **目标**: 在高性能数据面流水线中实现 GTP-U 协议头的剥离。
- **关键要求**: 支持多核心并行处理，确保在万兆带宽下保持低时延和高吞吐量，同时需兼容 3GPP TS 29.281 标准。

### 2.3 SR (System Requirement) - 系统规格
**编号**: `SR.UPF.FUN.001.001.001`
**规格详述**: **基于 VPP 节点的 GTP-U 报头剥离和 TEID 匹配**
- **实现逻辑**: 在 VPP 转发图 (Forwarding Graph) 中新增一个名为 `upf-gtpu-decap` 的核心节点。
- **功能细节**:
  1. **TEID 提取**: 从 GTP-U 头部提取 32 位的 TEID (Tunnel Endpoint Identifier)。
  2. **上下文匹配**: 根据 TEID 匹配本地 PDR (Packet Detection Rule) 中的隧道上下文，获取会话信息。
  3. **报头剥离**: 剥离外层 IP 报头、UDP 报头以及 8 字节的 GTP-U 报头。

### 2.4 AR (Allocation Requirement) - 分配需求
**编号**: `AR.UPF.FUN.001.001.001.001`
**分配模块**: **UPF-D (数据面 / VPP Plugin)**
- **详细描述**: VPP `upf-gtpu-decap` 节点核心逻辑的 C 语言实现。
- **实现要求**:
  - 利用 VPP 向量化处理能力，批量处理报文。
  - 调用 `vlib_buffer_advance` 接口精确调整缓冲区起始指针。
  - 维护高性能哈希表以实现 TEID 到 PDR 的快速查找。
  - 正确更新 `vnet_buffer` 元数据，确保后续转发流程（如下行路由或 PDR 匹配）能识别内层负载。

### 2.5 TC (Test Case) - 测试用例
**编号**: `TC.AR.FUN.001.001.001.001.UT.001`
**类型**: **UT (单元测试)**
**验证目标**: 验证 8 字节 GTP 报头移除的正确性及缓冲区指针的操作精度。
**测试步骤**:
1. **环境准备**: 构造一个 mock `vlib_buffer_t` (mbuf)，预填入包含外层 IP/UDP/GTP 报头和内层 ICMP 负载的二进制数据。
2. **操作执行**: 调用解封装处理函数 `upf_gtpu_decap_node_fn` 或其内部核心剥离逻辑。
3. **数据校验**:
   - 验证 `current_data` 指针是否已按预期后移 8 字节（对应 GTP 头长度）。
   - 检查 `current_length` 是否已同步减去相应的报头长度。
   - 对比缓冲区首地址偏移后的内容是否与原始 ICMP 负载的首字节完全一致。
4. **预期结果**: 缓冲区成功“剥离”外层封装，露出原始负载。
