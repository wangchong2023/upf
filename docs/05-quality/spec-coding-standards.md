# 5G UPF 代码编写规约速查表 (Cheat Sheet v1.0)

## 1. 基础命名 (Naming Convention)
| 对象 | 规范 | 示例 |
| :--- | :--- | :--- |
| **变量/函数** | `snake_case` | `pdu_session`, `get_buffer()` |
| **结构体/类型** | `snake_case` + `_t` | `session_context_t` |
| **宏/常量** | 全大写 + 下划线 | `MAX_GTP_PAYLOAD` |
| **全局变量** | `g_` + `snake_case` | `g_total_packets` |

## 2. 华为排版准则 (Layout)
- **大括号换行**: 左大括号 `{` 必须单独起一行。
  ```c
  if ((NULL == ptr))
  {
      do_action();
  }
  ```
- **括号完备性**: 逻辑表达式必须使用完备括号：`if ((a == b) && (c == d))`。
- **恒等风格**: 优先使用左常量比较：`if (NULL == p_buf)`。

## 3. C 语言安全红线 (Security)
- **初始化**: 变量声明必须赋初值：`uint32_t count = 0;`。
- **函数禁止项**: 严禁使用 `strcpy`, `strcat`, `sprintf`, `gets`。
- **函数推荐项**: 必须使用 `strncpy_s`, `strncat_s`, `snprintf_s`, `fgets`。
- **内存清零**: 敏感数据释放前强制 `memset_s`。

## 4. 高性能 VPP 规约 (Performance)
- **Dual-Loop**: 必须实现四报文循环处理以优化 SIMD。
- **Prefetch**: 处理当前报文时，必须预取后续报文 Header。
- **Zero-Copy**: 转发面严禁不必要的 `memcpy`。

## 5. 代码注释规约 (Commenting Standards)
### 5.1 C 语言 (Doxygen 风格)
- **文件头**: 必须包含 `@file`, `@brief`, `@author`, `@date`。
- **函数注释**: 必须包含 `@brief`, `@param`, `@return`, `@note` (若有特殊内存处理)。
- **逻辑注释**: 关键算法逻辑每 10-15 行代码需有一行说明。

### 5.2 Go 语言 (GoDoc 风格)
- **导出成员**: 所有导出的函数、结构体、接口必须有首行总结性注释。
- **包注释**: 每个包的 `doc.go` 或主文件中必须有包级说明。

### 5.3 脚本 (JSDoc 风格)
- **头部声明**: 必须包含规范的 JSDoc `@职责` 或 Shebang (`#!/bin/bash`)。
- **函数定义**: 必须使用 `/** ... */` 描述输入输出。

## 6. 脚本编码规约 (Script Standards - JS/Shell)
- **安全性**: 严禁使用 `eval()`，严禁在脚本中硬编码密钥或 Token。
- **健壮性**: JS 脚本必须包含必要的 `try-catch` 错误处理；Shell 脚本建议开启 `set -e`。
- **职责单一**: 一个脚本只负责一项核心治理任务，复杂任务应通过 Agent 进行编排。

## 7. 代码重构规约 (Refactoring Standards)
- **触发条件**: 
    - 圈复杂度 (Cyclomatic Complexity) > 15。
    - 重复代码块 (Code Duplication) > 20 行。
    - 单个函数长度 > 100 行。
- **管理流程**: 
    - 重构必须创建 `refactor` 类型提交，并关联 `IS.REFACTOR.xxx` 编号。
    - 重构前必须确保 100% 单元测试覆盖，重构后必须通过 `make quality-gate`。
    - 大规模重构 (涉及 5 个以上文件) 必须由 **ARCHITECT** 执行 TR2 架构评审。

## 8. Git 与提交规约 (Git & Commit Standards)
- **提交信息语言**: 所有的 Commit Message 必须采用 **中文** 编写。
- **格式规范**: 必须遵循 `类型(模块): 描述` 格式（参考 Angular 规范）。
    - 示例: `feat(治理): 实现自动化发布流水线`。
- **关联性**: 每个提交必须物理关联一个需求编号（AR.xxx）或缺陷编号（IS.xxx）。
- **分支原则**: 禁止直接向 `main` 分支提交代码，所有变更必须通过 PR 合入。

## 9. 质量准出 (Quality Gate)
- **TDD**: 先写失败的测试，再写实现代码。
- **ASan**: `make quality-gate` 必须 0 内存错误。
- **Linter**: `cppcheck`, `clang-tidy` 及 `mgr-script-audit` 必须 0 警告。
