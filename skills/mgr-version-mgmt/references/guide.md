# 5G UPF 版本管理与同步指南 (Expert Guide v1.0)

## 1. 版本的战略定位
| 序列 | 版本类型 | 命名范式 | 核心职责 |
| :--- | :--- | :--- | :--- |
| 1 | **基线版本** | `v1.0.x` | 确立 3GPP R16 基础转发面与 CP 交互契约。 |
| 2 | **特性版本** | `v1.x.0` | 承载切片、Token 治理等商业差异化特性。 |
| 3 | **架构版本** | `vX.0.0` | 涉及转发流水线重构或平台迁移（如 VPP 升级）。 |

## 2. 需求与版本的物理映射
- **RTM 标记**: 在 `docs/spec-rtm.md` 中，每一项 `AR ID` 的备注列需包含 `[Target: vX.Y.Z]`。
- **Agent 审计**: `agent-pm` 在刷新 RTM 时，自动校验 Target Version 字段是否存在。

## 3. 版本间同步 (Backport) 操作 SoP
当在 `main` (已发布版本) 发现 P0 缺陷并修复后，必须同步至 `develop`:
1. **定位提交**: `git log --grep="FIX"` 找到修复 Commit。
2. **切换分支**: `git checkout develop`。
3. **挑选合并**: `git cherry-pick <commit-hash>`。
4. **验证门控**: `make quality-gate` 确保同步后未破坏新版本逻辑。

## 4. 冲突解决准则
- **逻辑冲突**: 优先保留高版本的架构优化，手动重构补丁逻辑。
- **需求冲突**: 若低版本的补丁改变了高版本的 SR 契约，必须发起 RCR 变更流程。
