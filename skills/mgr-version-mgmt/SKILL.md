# 版本管理专家规范 (mgr-version-mgmt)

> **专家级工作流**: 版本需求对齐；多版本分支策略；版本间变更同步 (Backport)；版本健康度审计。

## 1. 基于版本的需求对齐 (Version-Requirement Alignment)
- **版本锚点**: 每一项 AR (分配需求) 必须映射至一个具体的交付版本（如 v1.0.0, v1.1.0）。
- **版本定位 (Positioning)**: 
    - **基线版 (Baseline)**: 核心 3GPP 转发逻辑与基础 OAM。
    - **增强版 (Enhanced)**: 行业专项特性 (如切片、Token 流量治理)。
    - **补丁版 (Maintenance)**: 仅限 P0/P1 缺陷修复与安全补丁。
- **冲突处理**: 若需求变更导致跨版本变动，必须执行 RCR (需求变更记录) 并由 ARCHITECT 评估架构兼容性。

## 2. 版本间变更同步 (Multi-Version Synchronization)
- **合入原则 (Merge Strategy)**: 
    - **Develop 分支**: 所有版本开发的首选汇聚点。
    - **Backport (反向合入)**: 在 `main` 或 `release/*` 分支发现的修复，必须立即执行 `make backport` 或手动合入 `develop`。
    - **Forward-port (前向合入)**: 低版本的通用改进应通过 cherry-pick 同步至高版本分支。
- **一致性审计**: 物理检查 RTM 矩阵在不同版本分支间的状态一致性。

## 3. 发布生命周期管理 (Release Lifecycle)
- **冻结期 (Code Freeze)**: 里程碑进入 TR5 后，`release/*` 分支进入冻结期，仅接受 `hotfix` 提交。
- **版本发布 (Physical Release)**:
    - 执行 `make release VERSION=vX.Y.Z` 自动生成 Tag 与 Changelog。
    - 验证 `quality-gate` 全量通过后方可执行 Git Push。

## 4. 指令集成
- **`make version-status`**: 查看各版本需求达成率。
- **`make version-sync-audit`**: 检查版本间关键修复是否已同步。
