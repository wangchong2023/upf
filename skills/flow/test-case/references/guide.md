# 用例与测试文档 (sr-test-case) 规范指南

## 1. 需求分层与测试类型映射
所有的测试用例必须基于需求标识（RR/IR/SR/AR）进行定义。

| 需求级别 | 需求类型 (简写) | 对应测试类型 | 验证点 |
| :--- | :--- | :--- | :--- |
| **AR (分配需求)** | FUN/PERF/SEC | **UT (单元测试)** | 模块函数、核心逻辑、性能微秒级抖动。 |
| **SR (系统规格)** | FUN/PERF/SEC | **IT (集成测试)** | 组件间通信、协议栈交互、子系统压力测试。 |
| **IR (特性需求)** | FUN/PERF/SEC | **ST (系统测试)** | 端到端业务流程 (PDU Session)、整体吞吐、安全审计。 |
| **RR (原始需求)** | FUN/MAINT | **AT (验收测试)** | 业务可用性、可维护性手册、故障恢复 SOP。 |

## 2. 测试用例命名规范
- **格式**: `TC.[需求标识].[测试类型].[序号]`
- **示例**: `TC.SR.UPF.FUN.001.001.IT.001`

## 3. 回归验证与结果记录
- 每次代码变更必须根据 RTM 触发对应的 AR (UT) 和 SR (IT) 回归。
- 所有的测试用例执行结果必须记录在追踪矩阵中。
