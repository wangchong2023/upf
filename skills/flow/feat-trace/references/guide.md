# 功能拆解与追踪 (IR/SR/AR) 规范指南 (IPD 标准 v0.3)

## 1. 命名与分级规范 (IPD 标准)
所有需求标识均采用大写，通过简写表明需求类型。

### 类型简写说明:
- **FUN**: 功能性需求 (Functional)
- **PERF**: 性能需求 (Performance)
- **MAINT**: 维护性需求 (Maintainability)
- **SEC**: 安全性需求 (Security)

### 分级结构:
- **IR (特性需求)**: `IR.项目名.类型.序列号`
  - 示例: `IR.UPF.FUN.001` (核心转发特性)
- **SR (系统规格)**: `SR.项目名.类型.序列号.子序号`
  - 示例: `SR.UPF.FUN.001.001` (GTP-U 封包规格)
- **AR (分配需求)**: `AR.项目名.类型.序列号.子序号.具体实现序号`
  - 示例: `AR.UPF.FUN.001.001.001` (分配给 UPF-D 的 GTP-U 解析代码实现)

## 2. RR -> IR 映射与拆解逻辑
### 映射方法论:
一个 `RR` 通常对应多个 `IR`。拆解时需考虑：
1. **控制面特性**: 该 `RR` 是否涉及信令处理 (N4)？(对应 `IR.XXX.FUN`)
2. **数据面特性**: 该 `RR` 是否涉及报文处理 (GTP-U/IP)？(对应 `IR.XXX.FUN/PERF`)
3. **管理面特性**: 该 `RR` 是否需要可观测性、配置上报 (OAM)？(对应 `IR.XXX.MAINT`)

### RR -> IR 映射矩阵示例:
| RR ID | IR ID | 特性名称 | 业务价值贡献 |
| :--- | :--- | :--- | :--- |
| RR.UPF.001 | IR.UPF.FUN.001 | PFCP 会话管理核心特性 | 实现 3GPP 标准信令终结。 |
| RR.UPF.001 | IR.UPF.FUN.002 | 会话状态监控特性 | 实现可观测性与异常告警。 |

## 3. 需求冲突分析与优先级排序
- **冲突检测**: 在 IR 层面识别是否存在功能重合（如两个特性都定义了 IP 查找逻辑）或逻辑矛盾。
- **优先级继承**: `IR` 应继承其父级 `RR` 的优先级，并在资源受限时根据业务价值进一步细化。

## 4. 追踪矩阵管理
- 每个 AR 必须能够唯一回溯到其父级 SR、IR 及根级 RR。
- 所有的代码实现和测试用例必须基于 AR/SR 进行关联。
